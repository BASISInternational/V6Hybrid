Purpose

The V6Hybrid project provides an environment for running both the current and legacy versions of AddonSoftware within the Barista MDI, and offers a sampling of modern functionality to those running Version 6 (V6), but not yet ready or able to make a wholesale conversion to Barista Addon, typically due to heavy customization in one or more modules.

Architecture

The project is available for download from BASIS, and is designed to install either directly into the BBj_home instance of BBj/Barista/Addon for demo/eval use, or into a separate Barista Addon instance, with some minor modification, for production use. It is structured as a Barista application, with standard V6 programs and associated demo data added into the standard Barista project file structure. New programs and reports are in the prog, reports, and sprocs directories, and modified V6 programs are in the DLR directory. The project runs under BBj alongside Barista Addon, so there is no need for a PRO/5 install. All of the V6 programs are in source format (rather than tokenized), and the demo data reflects a current timeframe (typically in March of the current release year). The demo hybrid assumes that Accounts Payable (AP) and General Ledger (GL) are running in Barista Addon, and the remaining modules are running in V6. All tasks are launched from a single, integrated menu. Several trigger files exist to provide access to Barista Addon's AP and GL data from V6 tasks such as SpeedSearch and Purchase Order Processing, as well as access to V6 module data as needed from within Barista Addon's AP and GL modules.

Installation

Obtain the V6Hybrid project from BASIS. Once you have the V6Hybrid zip file, you can install it in one of two ways:

  • Demonstration version
  • Extract the V6Hybrid project into the apps directory inside BBj_home (e.g., C:/bbj/apps/V6Hybrid).
  • Launch Demo Barista Addon, go to the Barista Utilities menu, and select Auto-synchronize Barista. Uncheck all applications except for the V6Hybrid project.
  • Production version
  • Use the AddonSoftware Installation Guide to create and sync up a new instance of Barista Addon in a different location, represented as <aon_prod> in the diagram below.
  • Launch the new instance. Dismiss the welcome message, accept the license agreement, and allow the Create SPROC Definitions task to complete. Close out of the new instance.
  • Copy/extract the V6Hybrid project into the <aon_prod> directory, and change the paths in the v6hybrid.syn file accordingly. In addition, add ALIAS lines as needed for users and printers (see Additional Notes later in this document for more information).
  • Re-launch the new instance and use the Barista Install Application Wizard to install the V6Hybrid project (navigate to <aon_prod>/V6Hybrid/config/v6hybrid.syn). The file structure for the new instance of Addon with the V6Hybrid project should look like Figure 1 below.

  documents/ (Present once a document is saved)

After installing V6Hybrid, log out and back into the new instance again so the auto-launch process can finish post-installation tasks specific to the V6Hybrid project. These tasks will create additional SPROC definitions, copy trigger definition and source files and change paths as needed, and disable the standard Addon menu, displaying only the "superset" V6Hybrid menu.

For versions prior to 22.00, navigate to the AddonSoftware Administration > V6 Administration > System Description Maintenance task. Call up each module that you'll be using, regardless of whether the module will run in V6 or Barista Addon, and change the Data Path and Program Path to reference the V6Hybrid project. With version 22.00+, the System Description records are updated automatically via an auto-launch task.

If you need to access V6 via the main CUI menu, as is sometimes the case during development and testing, then launch Barista Administration > Menus, select the V6Hybrid menu from the listbutton, then scroll to the bottom and expand the AddonSoftware Administration tree to reveal the V6 Administration > Utilities nodes. Select the Addon V6 CUI item, and alter the Config File Path to reference your install location (e.g. "<your_install_path>/V6Hybrid/config.aon"). Click the Save and then the Refresh button in the MDI toolbar, then close the menu. You will also need to examine the config.aon file located in the V6Hybrid folder. Change the V6Hybrid pathnames in the PREFIX to reference your install location (e.g., "<your_install_path>/V6Hybrid/ADATA", etc.).

If you used the AIUW to create a production instance of the project, you will need to copy in demo data for Barista/Addon if you wish to test. Copy the data files only (not sub-directories) from the BBj_home/apps/aon/data/ directory to your new data directory for Barista/Addon (this is likely a version neutral directory per the AIUW instructions). As you work toward going live you will be replacing the V6Hybrid demo data with live V6 data, and clearing the Barista/Addon demo data in order to run DataPort for the AP and GL modules. You'll also need to compare your production V6 programs with those in the V6Hybrid's ACT, DIS, MFG, etc. directories and modify or replace program files as needed to bring over any customizations you may have in V6.

At a Glance

At a high level, the V6Hybrid project demonstrates how to set up and run in a hybrid environment, where some tasks appear as CUI windows, and others as graphical forms, all launched from the Barista menu system and all running within the Barista MDI. One obvious benefit of this approach is that you can control access to all of the tasks using Barista Role-based Security.

The top panel of the V6Hybrid menu weaves the modules together seamlessly, moving from Barista Addon's Accounts Payable to V6's Accounts Receivable, etc.. In other words, the menu does not segregate the modules based on whether they're running in V6 or the current version. In the lower panel, each task for a given V6 module is defined individually. Launching a V6 task takes you directly into that CUI screen, and exiting the task dismisses the screen, leaving control with the Barista menu and bypassing the V6 menu altogether.

At a more detailed level, the demos included in the project showcase several areas where V6 users can leverage Barista Addon functionality while continuing to run portions of Addon in CUI.

  • Dashboard  –  The dashboard is a hybrid in itself, with widgets pulling information from either the current Addon database or V6 depending on the module.
  • A new demo widget on the Sales tab (Sales Comparison) uses SQL in the underlying SPROC (ARCMPSLS_BAR.prc) to query the V6 ART-03 table, then summarizes the results into the recordset used to populate the widget. The dashboard program populates two filters for the widget (customers, salespersons), the former using native reads based on a new key added to ARM-01, and the latter using another SQL query.
  • The remaining Sales widgets and the Company Aging widget are derived from their standard Addon counterparts, but modified to use the V6 database, while the GL widgets are populated from the current Addon database.
  • Individual widgets provide visual information at a glance.
  • The Aging/Sales Summary screen of the V6 Customer Maintenance task launches a pie chart widget depicting the customer's aged balance in addition to the traditional CUI display.
  • Queries  – Custom queries designed in Barista are called from V6 via Barista's Launch Task program to add powerful grid-based queries to CUI screens.
  • Cash Receipts customer number entry invokes a custom query on the V6 Customer Master file (ARM-01) in place of the V6 <F3> lookup.
  • Purchase Orders vendor number entry invokes a custom query on the Barista Addon Vendor Master file (apm_vendmast) in place of the V6 <F3> lookup.
  • DocOut reports  –  The DocOut reporting subsystem provides not only a convenient on-screen viewer, but also the ability to search, save, email and fax reports, as well as archive reports for later recall. *Most of the reports listed here also make use of a DocOutWrapper custom class to make it even easier to convert legacy print programs.

  1. Triggers

Triggers are perhaps the most critical component of the hybrid environment. Triggers are BBj source code files that are associated with a particular physical data file. Because triggers reside "below" the application code, they are invoked whenever an operation such as a read or write occurs on a given file, regardless of the application program involved. It is the triggers that provide access to Barista Addon's AP and GL data from programs running in V6, and likewise access to V6 data from the Barista Addon AP and GL programs. Understand that the V6Hybrid project does not function in a "parallel" fashion, where both databases are kept up to date. The AP and GL files are only updated in the Barista Addon database, and the other modules update only the V6 database.

Consider our hybrid environment, and the number of V6 programs outside of AP and GL that access AP or GL data: register/updates, SpeedSearch, lookups, Purchase Orders, Shop Floor, etc. Rather than modifying all of those V6 programs, we use triggers on the data files. With this approach, no V6 program changes are necessary. When a V6 program reads from or writes to an AP or GL file, the associated trigger code redirects the I/O to the Barista Addon file instead. Similarly, there are places in Barista Addon's AP module that access non-AP files, so triggers are used on these files to redirect the I/O to the V6 files.

A trigger consists of a .trigger definition file, which must reside in the same directory as the physical data file, and the associated BBj source code file. The location of the source code file is specified within the .trigger definition. For V6 triggers, the .trigger files are in the <install_loc>/V6Hybrid/ADATA/ directory, and the source code files are in  <install_loc>/V6Hybrid/ADATA/src/. Triggers needed for Barista Addon ship in the  <install_loc>/V6Hybrid/util/ folder. After installing the V6Hybrid project, a Barista Auto-launch task runs on the first launch to alter pathnames in the  <install_loc>/V6Hybrid/ADATA/*.trigger files relative to the install location, and to both move the Barista Addon *.trigger and associated source code files from <install_loc>/V6/util/ to the <install_loc>/aon/data/ and <install_loc>/aon/data/src/ directories, and change the pathnames accordingly.

Upgrades

*** DRAFT ***

Upgrades when running with the V6Hybrid are largely the same as any other Addon upgrade. Refer to the AddonSoftware Upgrade Guide for more information on the upgrade process in general. The main difference when using the V6Hybrid is that the "main" Addon upgrade is performed first, and the V6Hybrid project is re-installed once the Addon upgrade is completed.

The reason for this processing flow centers around the EM Update Wizard. Dictionary entries for several of the V6 tables are written to the Barista database when the V6Hybrid is installed. These definitions should not be present when running the EM Update Wizard. If the V6Hybrid definitions are in place during the EM Update Wizard you will likely encounter one or more of these issues: 

  • There are duplicate physical file names in the dictionary. For example, ARM_CUSTMAST references physical table (+ARDATA)arm-01, and the V6 definition for ARM01 references (+V6DATA)arm-01. Because the EM Update Wizard copies all files to a single destination directory, in a case like this, the first arm-01 will be copied and the other will produce an error and not be copied (because the first file is present and still open by the OCJ).
  • Some definitions for V6 tables in the dictionary may be incomplete or inaccurate, such as:
  • Definitions for certain record types in a non-normalized table. For example, there is a definition for ARM10F in order to allow Barista to query that table. That definition should not be used as part of a data copy, as it only represents one of the many record types in the file.
  • Definitions that are "shortcuts." For example, the IVM01 definition only describes the firm, item ID, and description, with the rest of the record as a "reserved" column. This allows Barista to query the V6 Item table for items/descriptions, without having to define in detail the entire record. Clearly this definition shouldn't be used for the data copy.

The basic flow for an Addon upgrade when using the V6Hybrid, then, is to complete the Addon upgrade on the TEST server, including copying data via the EM Update Wizard if applicable, and running post-upgrade conversions, and then install the V6Hybrid. Note that the V6Hybrid project should reside in a version-neutral location "above" AddonSoftware, as depicted earlier in Figure 1. As such, there should be no reason to make a new copy of the project during an upgrade. You will simply reinstall the same project into the upgraded Addon.

After installing V6Hybrid, use the Barista Utilities > Auto-Launch Maintenance form to change the Process Status back to "Incomplete" on the V6BLDSPROC, V6BLDTRGRS, and V6INITDEMO tasks, then exit and re-launch. When you re-launch, those Auto-Launch tasks will once again run to finish setting up the hybrid environment (these tasks can also be run individually from the AddonSoftware Administration > V6 Administration > Utilities menu).

  • Initialize the V6Hybrid Menu  –  The V6Hybrid menu is a replacement for the standard Addon menu. This task renames the Addon menu so the V6Hybrid menu gets loaded into the menu panel instead.
  • Create SPROC definitions  –  This task changes config file location for the V6Hybrid-related SPROCs to reference the "New" barista.cfg file.
  • Update Trigger definitions  –  This task changes the working directory and config file pathnames in the ADATA .trigger files to refer to the "New" Barista directory and barista.cfg file, copies the triggers stored in the V6Hybrid/util/triggers/ directory to the "New" <aon_prod>/vYY/app_data/ directory, and changes those pathnames as well.

After upgrading and testing on the TEST server, use the Barista Remove Applications utility to remove V6Hybrid from the new instance of Addon BEFORE packaging and then extracting "New" Addon back on the production server. Launch and perform a quick sync, then run the EM Update Wizard to copy data if necessary (to refresh data for major version upgrade), and finish the rest of the "go live" steps. Finally, reinstall the V6Hybrid and run the Create SPROC Definitions and Update Trigger Definitions tasks. The V6Hybrid menu should appear once V6Hybrid is installed and you restart Addon, however, you can also run the Initialize the V6Hybrid Menu task if needed.
